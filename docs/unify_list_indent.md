# 対訳ファイルのリストインデント

**ステータス: 現行の正書法 (0 スペース方式) を 2026/07/16 に採用**.
番号項目の下を一律 4 スペースにして語釈を項目の子にする統一は,
将来的な検討事項とする (下記). 2026/07/16 に実施した 0 スペース方式への
平準化は「関連改訂」参照.

## 現行の正書法 (2026/07/16 採用)

対訳ファイルのリストは次の形式を正とする.

- 番号付きパーリ文: 行頭の `1. ` (チャンクの通し番号)
- 語釈: 行頭 (0 スペース) の `* `
- サブ語釈 (複合語の構成要素などの補足): 2 スペースの `* `.
  さらに深い補足は 4 スペースの `* `
- 文全体の訳: 行頭 (0 スペース) の `→ 「...」`. 行頭の → は文の訳専用で,
  語や複合語への補足には使わない (サブ項目 `* ` で書く)
- リスト開始行と → 行の前には空行を入れる (docs/insert_blank_lines.md)

この形式は Markdown の方言に依存しないフラットな構造で, 番号の表示のみ
start 属性を反映するレンダラ (GitHub の GFM など CommonMark 系) に
依存する. CommonMark は規範ではなく, 表示先である GitHub の挙動を説明
するための参照仕様である. 例外はマーカー混在の dhp_356-359 と,
凍結ファイル (_v1)・履歴ファイル (_history.md).

## 前提となる CommonMark のルール

- あるブロックをリスト項目の「子」にするには, マーカー幅ぶんのインデント
  (= 内容カラム) が必要. `* ` なら 2, `1. ` なら 3, `12. ` なら 4
- リストマーカー自体の前の 0〜3 スペースは単に剥がされる

## 現行方式の構造のずれ (3 現象)

語釈リストの `* ` は行頭 (0 スペース) で, 番号項目の内容カラム (3〜4) に
届いていない. このため以下の現象があった. 現象 2・3 は 2026/07/16 の
平準化で解消し, 現行方式で残るのは現象 1 のみ:

1. 語釈リストが番号項目の子にならず, 各パーリ文は 1 項目だけの
   `<ol start="N">` に分断され, 語釈は兄弟の独立した `<ul>` になる.
   GitHub は start 属性を表示に反映するため番号は正しく見えるが,
   start を無視するレンダラでは全部 1 になる
2. 2 スペースの `  * ` はマーカー前インデントとして剥がされるだけで,
   col-0 の `* ` と同じトップレベルリストになる (字下げが効いていない).
   ただしその下のサブ語釈 (`    * `, 内容カラム 4 に一致) は正しく
   ネストしている
3. 2 スペースの `  →` 訳文行は直前リストのインデントで解釈が反転する.
   col-0 語釈の後なら最後の `<li>` 内の段落, 2 スペース語釈の後なら
   リスト外の独立段落になる

## 変換ツールでの実測 (2026/07/16)

sn/sn_54_2_07.md と dhammapada/dhp_378.md を HTML に変換して確認した.

- commonmark (公式リファレンス実装)・marked: `<ol start="N">` で番号を
  保持. サブ語釈のネストと行頭 → の独立段落も意図どおり
- kramdown (Jekyll / GitHub Pages の既定. 非 CommonMark): start 属性を
  出力しないため番号が全項目 1 になる. ネストと → 段落は意図どおり

md から HTML への変換ツールが CommonMark 系とは限らない (kramdown,
Python-Markdown, Pandoc の既定は独自方言). 現行方式の弱点は kramdown 系
での番号表示のみで, GitHub Pages (Jekyll) などで公開する場合が 4 スペース
統一 (下記) の実施タイミングになる.

## 将来的な検討事項: 番号項目の下を一律 4 スペースに統一

現象 1 (`<ol>` の分断) を解消するには, 番号項目の下を一律 4 スペースに
揃えて語釈と訳文を項目の子にする必要がある. 実施する場合は以下に従う.

```markdown
1. Idāni yaṃ taṃ bhagavatā ...

    * idāni = 「いまや、今や」
    * pasaṃsitvā = 「称讃して」（pasaṃsati の連続体）
        * （サブ語釈はさらに 4 スペース深く）

    → 「いまや、世尊が『…』とこのように称讃された…」

2. ‘‘ayampi kho, bhikkhave, ...
```

レンダリング結果 (marked で確認済み):

```html
<ol>
<li><p>Idāni yaṃ taṃ bhagavatā ...</p>
    <ul>
    <li>idāni = 「いまや、今や」</li>
    <li>pasaṃsitvā = 「称讃して」…
        <ul><li>（サブ語釈）</li></ul>
    </li>
    </ul>
    <p>→ 「いまや、世尊が…」</p>
</li>
...
</ol>
```

現状との違い:

1. `<ol>` が全体で 1 つに繋がり, 番号が start 属性に依存しなくなる
2. 語釈 `<ul>` が各項目の子になる (意図どおりの構造)
3. `→` 訳文行も項目内の段落になり, パーリ文・語釈・訳文が 1 つの
   `<li>` にまとまる. 表示上も項目のテキスト位置に揃って字下げされる

補足:

- `→` 行は厳密には 3 スペース (= `1. ` の内容カラム) で足りるが,
  語釈と揃えて 4 にするほうがソースが読みやすい. 余分な 1 スペースは
  HTML の空白折り畳みで表示に影響しない
- 4 スペースは Markdown.pl 系 (ネスト基準 4) とも互換

### 適用時の注意

- 対象は「番号付きパーリ文の直下」の語釈・サブ語釈・→ 行のみ.
  見出しや独立段落の下の語釈 (番号項目に属さないもの) は対象外の
  可能性があるため, 機械変換の前に文脈の分類が必要
- 変換はインデント量の変更なのでテキスト内容には触れないが,
  空行挿入 (追加のみ) と違い既存行の書き換えになる. 検証は
  「diff の全変更行が先頭空白のみの差か」を機械的に確認したうえで,
  文脈付き diff の目視を行う
- 適用したら対応する _history.md に履歴を追記する

### 実施を見送っている理由 (2026/07/15 判断)

- GitHub 上の表示は現状でもほぼ意図どおり (start 属性が効くため)
- 対象行数が多く, 既存行の書き換えを伴うため, 表示上の実益が小さい
  段階では実施しない

## 関連改訂 (2026/07/16)

上記の 4 スペース方式 (番号項目の内容カラムに揃える) とは逆に, 多数派で
ある「語釈 0 スペース, サブ語釈 2 スペース」の方式へ逸脱を平準化する
改訂を実施した.

1. **語釈インデントの統一**: 内容カラム方式で書かれていた
   dhp_002 の Opus ブロック (語釈 3〜4, サブ語釈 5 スペース) と,
   トップレベル語釈が 2 スペースだった 4 ファイル (dhp_025,
   mn_attha_061_with_paraphrase, sn_54_2_07, sn_54_2_08) を 0 / 2 スペースに
   統一
2. **文訳 → 行の字下げ解除**: 2 スペースで字下げされていた文全体の訳の
   → 行 145 箇所 (vism_08, patis_1_03, patis_attha_1_03) を行頭に統一
3. **語レベル注記のリスト項目化**: 直前の語釈 1 語の訳や複合語注記と
   して字下げされていた → 行 110 箇所の行頭マーカーを → から * に置換し,
   サブ語釈と同じ「インデント + *」の規則に統一. これにより字下げ →
   行の解釈反転 (現象 3) はほぼ解消し, 「リスト項目 = 語レベル,
   行頭の → 段落 = 文全体の訳」という区別になった
4. **字下げセクションの平準化**: patis_1_03 と patis_attha_1_03 に
   混在していた「語釈 2 スペース + 文訳 → 4 スペース」の字下げセクション
   (番号項目直下の語釈で計 598 項目) を 0 / 2 スペース方式に左シフト.
   領域単位の判定 (番号項目の直後に最初に現れるリスト行のインデントで
   判別) をスクリプトで行い, 通常セクションのサブ語釈には触れずに
   計 3,037 行をシフトした

## インデント機械変換の検証手順 (2026/07/24 知見)

インデントのみの変更でも Markdown ではリスト解釈 (ネスト・項目の帰属) が
変わりうるため, 機械変換を適用したら次の 3 段階で検証する.

1. 変更行の機械検証: 変更前 (HEAD) と行単位で突き合わせ, 全変更行が
   「先頭空白のみの差」であることをスクリプトで assert する
   (行数不変・本文差ゼロ)
2. レンダリング比較: 変更前後のファイルを CommonMark 系レンダラで
   HTML 化して diff し, リスト構造 (ol/ul/li のネスト, ol の start 属性)
   が同一であることを確認する. 実測は「変換ツールでの実測」と同じ
   marked を使う (`npx --yes marked -i in.md -o out.html`).
   段落内テキストの空白だけの差なら HTML の空白折り畳みにより表示等価
3. 対訳の内容検証: scripts/verify_taiyaku.rb の結果 (再掲行の判定と
   見出し検査) が変更前後で同一であることを確認する

レンダリング比較は変換と無関係の既存の構造問題も可視化する
(「N.」だけの行が空の番号項目 `<ol start="N"><li></li></ol>` になる
issue #3 はこの比較で発見された).
