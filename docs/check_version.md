# 校訂チェックの作り方 (指摘方式)

第一稿 (Gemini 等で生成した対訳) のうち, 校訂が必要そうな箇所だけを指摘する
軽量な md を作る手順をまとめる. 全文に出典をつける全文校訂方式や生成の自動化は
ここでは扱わず, 軽量な指摘方式に絞る.

## 目的

- 第一稿の全文を再掲せず, 判断が要る箇所 (誤り・解釈分岐・要確認) だけを集約する
- 確認済みで正しい訳の冗長な再掲を避け, 高い信号対雑音比を得る

## reviewed (全文校訂) との使い分け

- 指摘方式 (本書): 多くの経はこちらで足りる. 第一稿は概して精度が高く, 要修正は
  数点に留まることが多い (例: MN 61 は実質 1 件, MN 62 は高 1 件 + 中低 3 件)
- 全文校訂方式: 重要経・精読対象・多読時の参照辞書が欲しい経に
  限定する. 全語に出典がつくが分量が大きい (MN 61 で 1000 行超)

## 入出力

- 入力: `mn/mn_NNN.md` などの第一稿
- 出力: `mn/mn_NNN_check.md` (第一稿は保持)
- 例: `mn_062.md` -> `mn_062_check.md`

## 使う MCP

- pali-dict: Main / Concise / PTS 辞書. `search_pali_word`
- dop: A Dictionary of Pali (Cone, a-bh のみ). `lookup_dop`
- cscd-text: CSCD 三蔵 (本文 / 註 / 復註). 註釈引用と原文照合
- pali-text: PTS 原典 (補助)

## 手順

1. スキャン: 第一稿の対訳を通読し, 校訂候補を拾う. 教理・比喩・定型句に集中する
   (narrative は精度が高く候補が出にくい)
2. 候補の検証: 候補語を pali-dict / dop で引き, 必要なら cscd-text で aṭṭhakathā を
   引く. idiom・否定・命令の定型句は aṭṭhakathā を優先する (全文校訂方式の
   「修正の判断基準」と同じ)
3. 優先度付け: 高 (要修正/要検討) / 中 (註記推奨) / 低 (語の精度, 任意) に分ける
4. 指摘の記述: 箇所, 第一稿の訳, 指摘内容, 根拠 (辞書/註釈), 提案 を簡潔に書く
5. 範囲とネガティブ結果の明記: どこを重点確認しどこを流し読みしたか, および
   「確認したが問題なし」の箇所も書き, チェックの網羅度を読者に示す

## 並列サブエージェントによるスキャン

長い経 (例: vism_08 は 6000 行超) は, Agent ツールで複数のサブエージェントを起動し
行範囲を分担して並列にスキャンする. assāsa/passāsa の方向検出もこの方法を使った.

- 区間分割: 節境界で 5 区間程度に分け, 各エージェントに担当範囲 (行 a-b) を指定する
- model: opus を必ず明示する (校訂は最高モデルでないと意味がない)
- 各エージェントへの指示: 観点 (方向違反・語釈誤り・idiom・内部不整合・解釈混入),
  MCP のロード方法 (ToolSearch で pali-dict / dop / cscd-text), 根拠を取れた flag
  だけ報告, 出力フォーマット (高 / 中 / 低 + 確認したが問題なし)
- 並列にできるのは読み取り・スキャンのみ. 同一ファイルを並列で「編集」はできない
  (書き込み競合). 編集は主エージェントが結果を集約して行う

### 検証の併用 (重要)

サブエージェントの報告はそのまま信じず, 主エージェントが裏取りする.

- 範囲の過大一般化が起きる: 例「§220-223 が全違反」と報告されたが, 実際は §222
  (方向語なし) と §223 (複合語のみ) は該当しなかった
- 最高モデルでも個別の誤読は出る: 例 vīmaṃsati の語源を「正しい」と誤判定した
- 対策: 集約後に第一稿 (_v1) へ grep して節ごとに照合し, idiom は aṭṭhakathā を
  引き直す. assāsa/passāsa は「対訳の内部不整合の目視 → 註/辞書で確定 → 並列スキャンで
  網羅 → _v1 への grep で範囲確定」の順で検出・確定した

## 出力フォーマット

- ヘッダ: 第一稿のファイル名とモデル, チェックに使ったモデルと MCP, 日付,
  優先度の凡例
- チェック範囲: 重点確認した段落 / 流し読みした段落 / 原文照合の有無
- 指摘: 高 / 中 / 低 ごとに, 箇所・第一稿の訳・指摘・根拠・提案
- 確認したが問題なし: 一見要修正だが第一稿で正しい箇所. 字義に引きずられて
  誤修正しないための記録 (例: sasakkaṃ は PTSD 字義「できる限り」だが註釈で
  「決して」が正しい)
- メモ: 件数の要約と所見

実例は `mn/mn_061_check.md`, `mn/mn_062_check.md` を参照.

## 所見 (ヒューリスティクス)

- 候補が出やすいのは教理・比喩・定型句. 五大の比喩, 四梵住, 安那般那念のような
  技術的な箇所を重点的に見る
- 辞書字義と慣用句の意味は一致しないことがある. idiom は aṭṭhakathā で確定する.
  字義だけで第一稿を修正しない (全文校訂方式と共通)
- 「確認したが問題なし」を残すことで, 同じ語を次回また誤修正しかけるのを防ぐ
- 原文の byte 照合まで要るなら reviewed 方式に切り替える. 指摘方式は訳語の指摘に
  限定するのが軽量さを保つコツ

## スキル化に向けて

- 前提: pali-dict, dop, cscd-text の MCP が接続されていること
- Claude Code 本体 (対話 / エージェント) で実行する (MCP を使うため)
- 並列スキャンでサブエージェントを使う場合は, 各エージェントに必ず model: opus を
  明示する (校訂は最高モデルでないと意味がない)
- 引数の案: 対象ファイル, 重点確認する段落範囲, 出力先
- 第一稿を上書きしない (別ファイルに出力する)
- まず指摘方式で全体を見て, 要修正が多い / 精読対象の経だけ reviewed 方式に
  格上げする運用が効率的
