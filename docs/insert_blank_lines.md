# 対訳ファイルの空行スタイル改訂手順

初期の対訳ファイルには「パーリ文の直後に空行なしで語釈リスト, リストの直後に
空行なしで → 訳文行」というスタイルの箇所がある. これを空行ありのスタイルに
統一する手順.

```
1. Idāni yaṃ taṃ bhagavatā ...
* idāni = 「いまや、今や」
* pasaṃsitvā = 「称讃して」（pasaṃsati の連続体）
→「いまや、世尊が『…』と...」
```

↓ 改訂後

```
1. Idāni yaṃ taṃ bhagavatā ...

* idāni = 「いまや、今や」
* pasaṃsitvā = 「称讃して」（pasaṃsati の連続体）

→「いまや、世尊が『…』と...」
```

## 背景 (なぜ空行が必要か)

- CommonMark / GFM (GitHub) では, 箇条書き直後の空行なしの `→` 行は
  lazy continuation として直前のリスト項目に吸収され, 訳文が最後の語釈と
  同じ行 (同じ `<li>` 内) につながって表示される. これが実害の大きい方
- 空行なしのリスト開始は GitHub ではリストとして解釈されるが,
  Python-Markdown 系 (MkDocs など) ではリストと認識されず段落に連結される
- 空行を入れると親リストが loose 扱いになり行間がやや広がるが,
  構造上の問題はない

## 対象パターン (2 種)

1. リスト開始行 (`* ` など) の直前が, 空行でも箇条書きでもない行
2. `→` 訳文行 (インデント付き含む) の直前が非空行

`→` 行のインデントはファイルにより「なし」「2 スペース」が混在するため,
`^[ \t]*→` で両方に一致させる.

パターン 2 は当初「直前が箇条書きの場合」に限定していたが, 直前が平文・
数字項目・※注・補足の括弧書きでも同じ連結問題が起こるため,
2026/07/15 の一括適用で「直前が非空行なら挿入」に拡張した.

## 検出

適用前に件数と内容を確認する.

```sh
# パターン1: リスト開始行の直前に空行がない箇所
awk 'NR>1 && $0 ~ /^[ \t]*[*+-] / && prev !~ /^[ \t]*$/ && prev !~ /^[ \t]*[*+-] / \
  {n++; printf "%d: %s\n   -> %s\n", NR-1, prev, $0} {prev=$0} END{print n+0 "件"}' <file>

# パターン2: → 行の直前が非空行の箇所
awk 'NR>1 && $0 ~ /^[ \t]*→/ && prev !~ /^[ \t]*$/ \
  {n++} {prev=$0} END{print n+0 "件"}' <file>
```

あわせて適用前に以下を洗う (2026/07/15 の一括適用時の実績).

```sh
# → 行が連続する箇所 (空行で分離してよいか個別判断が必要)
awk '$0 ~ /^[ \t　]*→/ && prev ~ /^[ \t　]*→/ {print FILENAME" "NR} {prev=$0}' <file>

# 全角スペースでインデントされた語釈・→ 行 (半角 2 スペースに直してから適用)
grep -nE '^　+[*→+-]' <file>

# コードフェンス・スペース入り水平線 (- - -)・タブ・YAML フロントマターの有無
# いずれも 2026/07/15 時点の対訳ファイルには存在しなかった
```

## 適用スクリプト

各行を直前の行と突き合わせ, 該当パターンの前に空行を 1 行挿入する
ワンパスの awk. テキスト内容には触れない.

```sh
awk '
NR>1 && $0 ~ /^[ \t]*[*+-] / && prev !~ /^[ \t]*$/ && prev !~ /^[ \t]*[*+-] / {print ""}
NR>1 && $0 ~ /^[ \t]*→/ && prev !~ /^[ \t]*$/ {print ""}
{print; prev=$0}
' <file> > <file>.tmp
```

一時ファイルに出力して検証してから本体へ反映する. 複数ファイルへの
一括適用時は, ファイルごとに「行数の増分 = 検出件数 (+ 末尾改行補完)」を
assert し, 不一致なら反映せずに報告する.

## 検証

1. 行数の増分が検出件数の合計と一致すること (`wc -l`)
   - 元ファイル末尾に改行がない場合, awk が補完するため +1 になる.
     patis_1_03.md で実際に発生した. 補完は残してよいが, diff 上は
     末尾行の内容同一の書き換えとして見える
2. **git diff の変更行を機械的に分類**し, 追加がすべて空行で
   削除・変更がないこと (末尾改行補完を除く)
   ```sh
   git --no-pager diff --no-color <file> | grep -E '^[+-]' | \
     grep -vE '^(\+\+\+|---)' | sort | uniq -c   # 「N +」だけになるべき
   ```
3. 検出コマンドを再実行して両パターンとも 0 件になること
4. 変更箇所の数箇所 (冒頭, ネストした語釈, 補足段落つきの節など) を
   文脈付きで目視確認する

## 注意点

- 2 スペースインデント付きの `  →` 行は, 空行を入れても Markdown 上は
  最後のリスト項目内の独立した段落として扱われる (インデントが項目の
  内容位置に達しているため). 語釈と同じ行に連結される問題は解消される.
  完全にリスト外の段落にするにはインデント除去が必要だが, 訳文を
  一段下げる見た目の意図を尊重してインデントは保持する
- 語釈のネスト (`  * ...` や `    * ...`) の直前が箇条書きの場合は
  対象外 (パターン 1 の `prev` 条件で除外される). ネストに空行は入れない
- この改訂で解消するのは表示上の連結問題のみ. 語釈リストが番号項目の
  子になっていない (番号項目の内容カラム 3〜4 に対しインデントが
  0〜2 スペースで不足) 等の構造上のずれは残る. インデント統一は
  別途の改訂として保留中
- 適用後は対応する `_history.md` に履歴を追記する
