# 自動化以前の対訳ファイルへの後追い検証手順

自動化手順 (docs/automation.md) の整備以前に作成された対訳ファイルに,
同手順の検証を後から適用し, パーリ行を VRI 正本に整合させる手順.

自動化手順が新規生成のパイプラインであるのに対し, 本書は既存ファイルへの
検証と個別修正. スクリプト (extract_chunks.rb, verify_taiyaku.rb) は
読み取り専用で利用し, 生成・組み立てのステップは使わない. 作業の過程で
スクリプト側の拡張が必要になった場合 (hangnum 対応の例) は, 本書ではなく
automation.md 側の知見として記録する.

適用実績: visuddhimagga/vism_08.md と patisambhidamagga/patis_1_03.md
(いずれも 2026/07/24. 詳細は各 _history.md).

## 背景: 旧プロセス由来の混入

自動化以前のファイルは「チャンク化 (ブラウザからのコピー), Claude Web への
入力, 出力の md への貼り付け」を手動で行っていた. 原文段落は VRI テキストの
コピーで LLM の打ち直しではないため文字・語順の忠実度は高いが, 次の混入が
起きる.

- NBSP (U+00A0): tipitaka.org の HTML が &nbsp; を使うため, ブラウザからの
  コピーで段落番号の直後や異読注ブラケットの前後に混入する
- 対訳側 (LLM 出力) の再掲行ドリフト: 正本にないカンマの追加,
  文中小文字の大文字化, 綴りの標準化

## 手順

### 1. 正本の用意

VRI XML を _tmp/ に取得し, 対象範囲の plain text (source.txt) を作る.

- 経や単純な章は extract_chunks.rb でそのまま作れる (chunkspec は
  全段落を 1 グループにすれば十分. チャンクは使わない)
- 内部に subhead を含む篇 (Paṭis の title 配下など) は extract_chunks.rb
  が最初の境界で止まるため, 同じ to_plain 変換で範囲内の全段落 (subhead,
  centre の結び行も含む) を書き出す使い捨てスクリプトで対応する.
  extract_chunks.rb 本体はこの用途のために変更しない
- 原文ブロック照合用に subhead を除いた版も併せて出すと後段が楽になる
  (md は subhead を手動見出しにしていることがあるため)

### 2. 検証と分類

verify_taiyaku.rb をかけ, NG を「機械的な混入」と「意図的な編集」に
分類する. 分類には NG 行と正本の対応箇所を空白・引用符除去レベルで
突き合わせて分岐点を出すと速い.

機械的な混入 (修正する):

- NBSP: 通常スペースに一括置換. 事前に和文に隣接する NBSP がないことを
  確認し, 置換が「NBSP -> スペースのみ」であることを文字単位で assert する
- 段落番号直後の空白なし: VRI 原文の表記 (219.Sosatova のような形) なので,
  自動化の規約どおり空白を 1 個補う
- 再掲行の標準化ドリフト: VRI の形に戻す. VRI 側の誤植が疑われる綴りは
  他版 (PTS) や並行箇所 (cscd-text / pts-text MCP) で確認したうえで
  VRI の形に戻し, 事実を語釈の注記に書く (例: vism_08 の sīghaṃ sīyaṃ,
  āhiṇḍahi)
- 「N.」だけのラベル行: 空の番号項目としてレンダリングされるため,
  二重番号方式 (`1. 220. Sabbakāyapaṭisaṃvedī ...`) に統合する

意図的な編集 (NG のまま保持してよい. 迷えばユーザに確認):

- 日本語の注記 (〔第三歩〕, 〔異読：〕, （偈・第一句）, ※ など)
- 連声・複合語の教育的な分解 (pan' ettha, ti-ādim āha, paṇidhāya ti)
- 長文の中略記号 (…) による分割・省略, 反復句の圧縮
- 偈の半行連結に伴う行頭大文字の小文字化
- 手動で入れた見出し (NG-HEADING になるが保持)
- 引用符の形. 対訳部分の ASCII の "..." や全角の “...” による語句引用は
  編集上の表記であり, VRI の ‘‘ ’’ へ機械置換しない (2026/07/24 に
  patis_1_03 と patis_attha_1_03 で置換したが編集意図に反したため
  差し戻した. 引用符の統一を行いたい場合は事前にユーザに確認する).
  verify_taiyaku.rb は OK-NOSPACE の除去対象に " と “ ” を含めており
  (2026/07/24 追加), これらの編集表記は NG にならない.
  置換しないのは対訳部分に限る. 原文ブロックは byte-exact 原則の
  とおり VRI の ‘‘ ’’ のままであるべきで, もし原文側に ASCII 引用符が
  あればそれは修正対象の混入

### 3. 原文ブロックの照合

原文ブロック (## 見出しから ### 対訳 まで) を正本の段落列と突き合わせ,
byte-exact で一致することを確認する.

- 段落の対応付けは LCS で整列すると, 編集注 (※) の挿入や段落の分割が
  あってもずれずに差分だけが残る
- セクション境界で正本の 1 段落が分割されている場合は, 連結して正本と
  一致することを確認したうえで編集判断として保持する

### 4. 全体検証

- HEAD との全差分を機械分類し, 意図した種類の変更のみであることを
  確認する
- レンダリング比較 (marked で前後を HTML 化して構造 diff) を行う.
  手順の詳細は docs/unify_list_indent.md の「インデント機械変換の
  検証手順」参照. 空白補いで段落番号行が `<p>` から `<ol>` に変わるのは
  他の番号付き段落と同じ既知の形式なので問題ない
- 対応する _history.md に記録する

## 作業上の注意: Ruby ワンライナーによるファイル切断事故

本手順の一括修正 (NBSP 置換, 空白補いなど) を Ruby で行う際の注意.
実際に vism_08 と patis_attha_1_03 への適用時に同一の事故が 2 度起きた
(いずれも復元済み).

- ロケールが C のシェルでは, `ruby -e` のワンライナーにマルチバイト文字
  (パーリ語・日本語) を含めると US-ASCII として解釈され, SyntaxError や
  実行時の ArgumentError (invalid byte sequence in US-ASCII) になる
- 特に `ruby -i` (in-place 編集) は行を処理しながら対象ファイルを
  上書きするため, 実行時エラーで途中終了すると書きかけのまま切断される
  (vism_08 で 455KB -> 91KB に切断. patis_attha_1_03 で再発)
- 対策:
  - マルチバイトを扱う Ruby はワンライナーにせず, `# encoding: UTF-8` を
    付けたスクリプトファイルとして実行する
  - in-place 編集 (-i) は使わず, File.read / File.write で明示的に処理し,
    書き込み前に置換結果を assert する (差分が意図した種類のみか,
    サイズ・行数が想定どおりか)
  - やむを得ず -i を使う場合は必ず `-i.bak` でバックアップを残す
- 事故時は直ちに `wc -c` でサイズを確認し, git (または .bak) から復元して
  スクリプトで再適用する. 復元後は検証を全てやり直す

## 自動化手順との違い (重複させないための整理)

- 原文ブロックの byte-exact 化: 自動化はスクリプトが正本から書き込むこと
  で保証する. 本手順は既存の貼り付けテキストを正本と照合して直す
- 再掲行の NG: 自動化は生成のたびに verify で検出して戻す. 本手順は
  一度だけ全行を洗い, 意図的な編集は NG のまま残すことを許容する
- チャンク分割・生成・組み立て (automation.md の主要部) は本手順では
  一切行わない
